# v0.9.3 — measure home-table auto-resolution

## Bug fixed

`pbir_bulk_bind` / `pbir_add_visual` / `pbir_update_visual_bindings` (any code path that writes `type: "measure"` bindings) silently produced broken visuals when the LLM specified a fact-table entity for a measure that lived on a different table. Example: writing `Sales[Total Revenue]` for a measure authored on `_Measures` — Power BI Desktop accepted the file, the visual showed the measure name in its field well, but no data rendered because `SourceRef.Entity = "Sales"` pointed at the wrong table.

## Fix — two halves

1. **Validator suggests the home table.** When `validateFieldSpec` returns `measure_not_found` for a measure spec, it now walks every other table and surfaces matches as top-priority suggestions:
   - Single home → `"_Measures[Total Revenue] (actual home table)"` as the first suggestion.
   - Multiple homes → all listed as `"<table>[<measure>] (candidate home table)"`.
2. **Auto-resolver in `parseFieldSpec`.** When given an inventory and the measure name lives in **exactly one** other table, the entity is auto-corrected to that table and the rewrite is recorded. Tool handlers drain the per-call collector and attach `bindingAutoCorrections[]` to the response so the LLM sees the fix:

```json
{
  "success": true,
  "bindingAutoCorrections": [
    { "from": "Sales[Total Revenue]", "to": "_Measures[Total Revenue]", "reason": "measure home table" }
  ]
}
```

   Ambiguous (>1 home) cases are NOT auto-corrected — they surface as a normal `measure_not_found` validation error with both candidates in the suggestions. Columns and aggregations are never auto-corrected: those are physically bound to a table.

## Surface area

- `parseFieldSpec(spec, inventory?)` — new optional second arg. Backward-compatible: callers without inventory get the prior behaviour.
- `runBindingValidation(...)` returns `inventory` on the `ValidationOutcome` so handlers can thread it into the apply path.
- `createAndSaveVisual(project, pageId, spec, baseZ, inventory?)` — new optional 5th arg.
- `applyBindingsToVisual(visual, bindings, { autoFilters, inventory? })` — new optional `inventory` field on options.
- `beginBindingAutoCorrections()` / `drainBindingAutoCorrections()` — module-level collector helpers exported from `helpers/createVisual`.

Even when `MCP_BINDING_VALIDATION=off` is set, `runBindingValidation` now still loads the inventory so the auto-resolver can run — the spirit of "off" is silent-correct, not silent-fail. When the model can't be located (live-connect, missing sibling `.SemanticModel`), behaviour is unchanged from v0.9.2.

## Audit

Patched paths confirmed routing measure binds through inventory-aware `parseFieldSpec`:

- `src/tools/visuals.ts` `pbir_add_visual` (uses `createAndSaveVisual`)
- `src/tools/bindings.ts` `pbir_update_visual_bindings` (uses `applyBindingsToVisual`)
- `src/tools/bulk.ts` `pbir_bulk_bind` (uses `applyBindingsToVisual` per entry)
- `src/tools/layoutGrid.ts` `pbir_layout_grid` commit mode (uses `createAndSaveVisual`)

Direct `measureRef(...)` callers: only one — inside `parseFieldSpec` itself. No other site constructs `Measure` refs through the helper.

`src/tools/format.ts` `pbir_set_visual_sort` inlines its own `Measure` ref construction and does not run binding validation. Sort operations target an existing visual's measure references, not new bindings — out of scope for this PR but flagged for future work if the LLM also tries to set sort on a measure that needs home-table resolution.

## Tests
