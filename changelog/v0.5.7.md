# v0.5.7 — Skill Content Rewrite to 100% Coverage

**Date:** 2026-04-15

The second half of the parity push that started in v0.5.6. Where v0.5.6 wired `skills/` into the runtime and built the audit baseline, v0.5.7 rewrites the eight stale skill files against the current tool surface and drives missing coverage from 17 to 0.

## The Problem (continued from v0.5.6)

After v0.5.6 the audit reported:

```
Tools registered:   55
With skill mention: 38
Missing coverage:   17
Coverage:           69.1%
```

Eight skill files were stuck at v1.0 (2026-04-09) and described tools that had since changed shape, lost arguments, or been overshadowed by better one-call replacements:

- `slicers.md` — missing `multiSelect`/`slicerMode` from v0.5.5
- `visuals.md` — missing `change_visual_type`, `delete_visual`, `get_visual_types`, `bulk_delete_visuals`, batch mode, inline `containerFormat`/`visualFormat`/`dataColors`
- `formatting.md` — missing `set_conditional_format`, `set_visual_sort`, `format_visual` auto-routing, `categoryEntity`/`categoryProperty` on `set_datapoint_colors`
- `filters.md` — missing the `advanced` filter type with And/Or compound operators
- `pages.md` — missing `set_page_background`, `set_filter_pane`, `set_visual_interaction`, `get_page_summary`, `manage_extension_measures`, tooltip + drillthrough page types on `create_page`
- `themes.md` — missing `diff_report_theme`, `list_report_themes`, `audit_theme_compliance`, the new `get_report_theme` shape
- `report.md` — missing `model_usage`, `set_visual_interaction`, `set_filter_pane`, `load_tools`, `manage_extension_measures`, the 12-default / 42-on-demand tool loading model
- `token-usage.md` — pre-v0.4.9 numbers from before smart tool loading; no `bulk_*`, no `model_usage`, no `load_tools` guidance

## The Rewrite

Every file was rebuilt from the actual TypeScript source (`src/tools/*.ts` + `src/model-usage.ts` + `src/index.ts`) rather than from the previous prose. Each rewrite:

1. Starts with a "When to use" / "Tool surface" table listing every tool the file covers
2. Documents every tool argument with a real JSON example
3. Bumps the doc-version frontmatter to `2.0`
4. Cross-references sibling skills with `skills/<name>.md` links

## Audit Result

```
Tools registered:   55
With skill mention: 55
Missing coverage:   0
Coverage:           100.0%
```

All 55 registered tools now have at least one backtick-wrapped mention in `skills/*.md`. The audit script (`scripts/audit-skill-coverage.js`) runs clean with `--strict` and is ready to be wired into a pre-commit / CI hook in v0.5.8.

## What Each Rewrite Adds

### `skills/slicers.md` v2.0
- Four-slicer-type table (`slicer`, `listSlicer`, `textSlicer`, `advancedSlicerVisual`)
- `slicerMode` (Dropdown vs Basic) — only on the standard `slicer` type
- `multiSelect` parameter with full PBIR `singleSelect` inverse-semantics table
- `get_visual` slim mode detection rules — including PBI default fallbacks per type
- Inline formatting examples and a horizontal mixed-type slicer row in batch mode
- Common pitfalls (don't set `slicerMode` on non-`slicer` types, the `singleSelect` inversion trap)

### `skills/visuals.md` v2.0
- Full tool surface: `add_visual` (single + batch), `get_visual`, `list_visuals`, `get_visual_types`, `move_visual`, `duplicate_visual`, `change_visual_type`, `delete_visual`, `bulk_delete_visuals`
- 40+ visual type reference table mapping user phrasing → `visualType`
- Bucket names per visual (including the combo-chart `ColumnY`/`LineY` and scatter `Details`)
- `Table[Column]` field shorthand + verbose form + aggregation function list
- Container-only visuals: `image`, `actionButton`, `pageNavigator`, `shape`, `textbox`
- Inline formatting branches (`containerFormat`, `visualFormat`, `dataColors`) at create time
- `change_visual_type` caveats (bucket-compatible swaps only)

### `skills/formatting.md` v2.0
- Two-layer model: `visualContainerObjects` vs `objects`
- `format_visual` auto-routing with the explicit `CONTAINER_CATEGORIES` set
- Forced-target override (`target: "container"` / `"visual"`)
- Property encoding table (hex, boolean, number, string)
- Common container + visual category catalogues
- `set_visual_title` field-by-field
- `set_datapoint_colors` — both metadata mode (Series-based) and data-selector mode (`categoryEntity`/`categoryProperty` for category-based bar/column/pie/donut/treemap)
- `set_conditional_format` — rules mode (with `ComparisonKind` 0–5), gradient mode (linearGradient2/3), clear mode
- `set_visual_sort` — column / measure / aggregation, direction, `isDefaultSort`
- `apply_theme` four-preset list and what it skips

### `skills/filters.md` v2.0
- Filter types matrix with scope (page-only vs visual-only)
- `categorical` with optional pre-selected values
- `topN` with explicit "visual scope only" enforcement
- `relativeDate` with all five periods and the days/weeks/months/quarters/years cheatsheet
- `advanced` with all 12 operators, And/Or compound conditions, the unary `IsBlank`/`IsNotBlank` shape
- Filter vs slicer decision table

### `skills/pages.md` v2.0
- Page tools, page chrome tools, and bookmark tools split into three surface tables
- `create_page` standard / tooltip / drillthrough variants — including the auto-applied tooltip defaults (320×240, ActualSize, hidden) and the drillthrough `isAllFilter` mechanism
- `set_page_background` with canvas + wallpaper + clear semantics

- `manage_extension_measures` add/list/remove with the auto-delete-when-empty safety note
- Bookmarks: `list_bookmarks`, `add_bookmark`, `rename_bookmark`, `delete_bookmark`

### `skills/themes.md` v2.0

Nothing breaks. Every tool still works exactly as before. What changes:

- `guide("slicers")`, `guide("visuals")`, `guide("formatting")`, etc. now return doc that matches the actual tool surface
- LLM agents that read the guide as part of their bootstrap should produce noticeably better tool calls — fewer wrong arguments, more use of the cheap one-call patterns
