# v0.9.4 — Loose ends

Two consistency / regression fixes; no new tools, no artifact changes, no breaking surface.

## Fix 1 — `pbir_set_visual_sort` uses inventory-aware measure resolver

The v0.9.3 audit flagged that `pbir_set_visual_sort` (in `src/tools/format.ts`) built `Measure` `FieldRef` objects inline rather than going through `parseFieldSpec`. That's the same bug class as the original v0.9.3 binding fix, just at a different code site: an LLM passing `Sales[Total Revenue]` for a measure that actually lives on `_Measures` would write a sort definition with the wrong `SourceRef.Entity`, and Power BI Desktop would silently drop the sort.

The handler now:

1. Resolves the model inventory via `getInventoryForProject(project, resolveValidationMode(undefined))`.
2. Calls `parseFieldSpec({ field, type, aggregation }, inventory)` per sort entry — same path as `pbir_add_visual` / `pbir_update_visual_bindings`.
3. Drains `bindingAutoCorrections` and surfaces them on the response payload when any auto-correction fired.

Behaviour matches the rest of the v0.9.3 surface: ambiguous home tables (>1 candidate) are not auto-corrected, columns and aggregations are never auto-corrected, and live-connect / missing-`.SemanticModel` reports get the legacy pass-through behaviour.

## Fix 2 — finish `fail()` migration in `calculations` / `filters` / `format`

The W1.1 cleanup (commit `6fe4d54`) migrated inline `JSON.stringify({ success: false, ... })` patterns to the `fail()` helper from `src/helpers/mcpResult.ts` but explicitly left three files: `src/tools/calculations.ts` (2 patterns), `src/tools/filters.ts` (4 patterns), `src/tools/format.ts` (4 patterns). All 10 are now migrated. Behaviour preserved — `fail()` emits the same dual-channel envelope (`content[0].text` + `structuredContent` + `isError: true`) the inline patterns produced.

Final grep on master returns zero hits:

```
grep -nE 'JSON\.stringify\(\{\s*success:\s*false' src/tools/calculations.ts src/tools/filters.ts src/tools/format.ts
```

## Known follow-ups (flagged, not fixed)

Two other sites still construct `Measure` refs inline rather than going through `parseFieldSpec`. Both currently take an explicit `entity`+`property` rather than a `field` shorthand and are scoped to one-off targets, but mirror the same bug class at lower severity:

- `src/tools/format.ts:284` — `pbir_set_conditional_format` builds the conditional fieldExpr inline.
- `src/tools/filters.ts:87` — `topN` filter `orderBy` builds the order-by Measure inline.

Worth folding into a future cleanup pass if either ever takes a measure on the wrong table in practice.
