# v0.9.1 — Fix pbir_set_report -32602 outputSchema rejection

_Released 2026-04-27_

## Highlights

- **Bug fix: `pbir_set_report` was unusable in strict MCP clients** (most
  visibly: Claude Code). Calling it returned the protocol-level error
  `-32602: Structured content does not match the tool's output schema:
  data must NOT have additional properties`, and the bind side-effect
  never committed — `pbir_get_report` afterward still reported
  `"No report connected"`. Same shape on every mutation tool whose handler
  returned anything beyond `{success, error}` and was not in the per-tool
  `READ_TOOL_SCHEMAS` override map.

- **Root cause.** `GENERIC_OUTPUT_SCHEMA` at `src/index.ts:272` was a flat
  zod shape `{ success: z.boolean(), error: z.string().optional() }`. The
  MCP SDK serialises a flat ZodRawShape into a strict JSON Schema with
  `additionalProperties: false`, which rules out every handler payload
  with extras. The preceding comment block claimed the schema permitted
  any extra keys via `.passthrough()` — it did not. The eval suite missed
  this because evals bind via the `PBIR_REPORT_PATH` env var and never
  call `pbir_set_report`.

- **Fix.** Mutation tools and verbose-dump reads now opt out of structured-
  output validation entirely — `GENERIC_OUTPUT_SCHEMA` is `undefined` and
  `doRegister()` passes no `outputSchema` to the SDK for those tools. They
  still emit `structuredContent` (the dual-emit invariant in
  `helpers/mcpResult.ts` is unchanged); clients just don't validate it.
  Read tools listed in `READ_TOOL_SCHEMAS` keep their handcrafted schemas
  (every one already uses `.passthrough()` correctly). Comment block
  rewritten to match the code.

- **Bonus.** `pbir_get_visual_types` was hitting the same regression on
  the eval fixture (same generic-schema rejection class). It now passes
  alongside `pbir_set_report`.

- **Regression test.** New `scripts/test-set-report.js` (9 assertions)
  spawns `dist/index.js`, inspects the published `outputSchema` for both
  affected tools (asserting `additionalProperties !== false`), exercises
  `pbir_set_report` against the eval fixture and a bogus path, verifies
  the side effect via `pbir_get_report`, and confirms the dual-emit
  invariant. Wired into `test:all` as `npm run test:set-report`. Fails on
  the pre-fix build (catches `additionalProperties: false` on the
  published schema), passes on the fix.

- **Surface test relaxed.** `scripts/test-tool-surface.js` no longer
  requires every tool to publish an `outputSchema` — instead it parses
  `READ_TOOL_SCHEMAS` from `src/helpers/outputSchemas.ts` and asserts
  every read tool listed there does publish one. 14 read tools opt in;
  42 mutation/dump tools opt out. `test:all`, `audit:strict` (still 100 %),
  and the new `test:set-report` all green.

## Pre-release

The bug only manifests in strict MCP clients that validate
`structuredContent` against the published `outputSchema`. The new
regression test catches it at the schema level (the most reliable
indicator) but doesn't require a live strict client to run. Before
announcing v0.9.1, run `npm run eval` against Opus as a release-gate
sanity check — Anthropic API key required (`ANTHROPIC_API_KEY` env
var, ~13 minutes, ~$3). Eval should remain at 10/10.
