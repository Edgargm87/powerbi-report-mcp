# v0.5.9 — Audit CI Gate + Pre-Commit Hook

**Date:** 2026-04-15

Ships the CI gate deferred from v0.5.6/v0.5.7/v0.5.8: `audit-skill-coverage.js --strict` and the wireframe-validator suite now run automatically on every commit (via a pre-commit hook) and on every push/PR (via GitHub Actions). Also extracts the `DEFAULT_TOOLS` set into its own file so the runtime and the audit script read from one source, and teaches the audit to report default vs on-demand tiers separately.

## The Problem (continued from v0.5.8)

v0.5.6/v0.5.7 built the audit baseline and drove coverage to 100%. But the gate was still manual: nothing stopped someone from merging a new tool without a matching skill mention. The coverage metric would silently slide, the agents would slowly start failing on the new surface, and we wouldn't notice until the next audit run — exactly the regression loop we were trying to close.

Meanwhile the 12-default tool list lived as a local `const` inside `src/index.ts` (lines 24–41), which meant the audit script couldn't tell a default tool (loaded into every session, seen by every LLM) from an on-demand tool (only loaded on request). A missing default-tool mention is a much bigger deal than a missing on-demand mention, and the audit couldn't flag the difference.

## The Change

### Single source of truth for `DEFAULT_TOOLS`
- **NEW: `src/default-tools.ts`** — exports `DEFAULT_TOOLS: ReadonlySet<string>` with the 12 core tools. Imported by `src/index.ts` (so the runtime reads it) and parsed by `scripts/audit-skill-coverage.js` (so the gate reads it). No duplication.
- `src/index.ts` now imports instead of declaring — removes 17 lines of inline constant from `index.ts`.

### Audit tiers default vs on-demand
`scripts/audit-skill-coverage.js` now reads `src/default-tools.ts` and categorizes every tool as `default` or `on-demand`. Output shows:

```
Tools registered:   55
With skill mention: 55
Missing coverage:   0
Coverage:           100.0%

  Default tools:    12/12  (100.0%)
  On-demand tools:  43/43  (100.0%)
```

Default tools are tagged `[DEFAULT]` in the covered/missing lists. If any default tool falls out of coverage, the audit prints an extra warning explaining why that's worse than missing an on-demand tool ("these are the tools loaded into every session and seen by every LLM client"). JSON output now includes `summary.default` and `summary.onDemand` blocks for programmatic consumers.

### Pre-commit hook
- **NEW: `.githooks/pre-commit`** — shell script that runs `audit:strict` and `test:wireframe`. Skips on docs-only commits (only triggers when `src/`, `skills/`, `scripts/`, `package.json`, or `.githooks/` is touched).
- **NEW: `scripts/install-hooks.js`** — sets `core.hooksPath` to `.githooks/`. Runs `git config` and `chmod 755` (the chmod is a no-op on Windows, where git-for-windows doesn't need the exec bit).
- **NEW: `npm run hooks:install`** — one-shot installer. Documented in CONTRIBUTING.md as a setup step.

Skip in emergencies with `git commit --no-verify` — but CI runs the same gates on push, so a broken commit won't survive the PR.

### GitHub Actions CI
- **NEW: `.github/workflows/ci.yml`** — runs on push/PR to `master` / `main`:
  1. Checkout
  2. Setup Node 20 with npm cache
  3. `npm ci`
  4. `npm run build`
  5. `npm run audit:strict`
  6. `npm run test:wireframe`

Concurrency-cancels older runs for the same ref. Total workflow time is small — the audit is pure Node and the validator is pure Node, no network, no fixtures.

### New npm scripts
```json
"audit":          "node scripts/audit-skill-coverage.js"
"audit:strict":   "node scripts/audit-skill-coverage.js --strict"
"test:wireframe": "node scripts/test-wireframe-validator.js"
"test:all":       "npm run build && npm run audit:strict && npm run test:wireframe"
"hooks:install":  "node scripts/install-hooks.js"
```

`test:all` is the local equivalent of the CI job — one command to run everything a reviewer cares about.

### CONTRIBUTING.md rewrite (v1.2 → v1.3)
Three sections rewritten:
- **Setup** — adds `npm run hooks:install` as a setup step with a pointer to the gate explanations
- **Step 4 (default vs on-demand)** — points at `src/default-tools.ts` as the single source of truth; reiterates the "don't duplicate the list" rule
- **Step 5 (documentation)** — now marked **MANDATORY — enforced by CI**. Every new tool must have a backtick mention in a `skills/*.md` file or the audit will fail the pre-commit hook and CI. Default vs on-demand tools get slightly different guidance on what a "meaningful" mention looks like.
- **Section 6 (QA Expectations)** — adds a new "Automated gates" subsection with a table of the three gates (build, audit, validator), the commands, and what each checks. Manual QA unchanged.

## What Changes for Users

- **Contributors** must run `npm run hooks:install` once after cloning. Without it, the pre-commit gate is bypassed locally (CI will still catch any regression on push).
- **New-tool PRs** that don't add a backtick mention in a `skills/*.md` file will be automatically rejected by CI. This is the point of the gate — documentation drift is now a build failure, not a "we'll fix it later" note.
- **Default-tool changes** (adding/removing from `DEFAULT_TOOLS`) are now a single-file edit in `src/default-tools.ts` — no two-location sync.
- **MCP clients** connecting to the server will now see the correct version string (`0.5.9`) in the handshake instead of the stale `0.5.2`.
