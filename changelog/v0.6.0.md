# v0.6.0 — Skills & Safety Polish

**Date:** 2026-04-15

v0.6.0 is a content and safety release. No new tools, no schema changes. Three behaviour tightenings (bulk-ops confirmation gate, cross-row column-alignment warning, version bump) plus three new or expanded skill docs (`calculations`, `themes-per-visual`, mental-model prelude for `report-design`) and a DAX-authoring pointer in `svg-visuals`.

The theme: every change here is something an agent will *do differently* in the next session — either because a new guardrail catches a mistake, or because a skill file teaches the mental model that prevents the mistake in the first place.

## Behaviour changes

### Bulk-ops safety gate (`src/tools/bulk.ts`)

`bulk_delete_visuals`, `bulk_update_format`, and `bulk_bind` now enforce a 5-visual threshold. If the call would touch more than 5 visuals and `confirmBulk` is not `true`, the tool returns a structured error instead of running:

```json
{
  "success": false,
  "error": "Safety gate: this call would delete 9 visuals (threshold 5). Set confirmBulk: true to proceed, or reduce the list to ≤5 items.",
  "count": 9,
  "threshold": 5,
  "confirmBulkRequired": true
}
```

Why: the common agent failure mode is "list every visual on a page, pipe the id array straight into `bulk_delete_visuals`, wipe the page". The gate forces a second thought without blocking legitimate large operations. All three bulk tools gained a `confirmBulk: z.coerce.boolean()` parameter and a single shared `bulkSafetyError()` helper.

Documented in `skills/visuals.md` with a dedicated "Bulk safety gate" section. Explicit rule for the agent: **Never set `confirmBulk: true` reflexively — the gate exists to force a second thought.**

### Wireframe validator: cross-row column drift (`src/wireframe-validator.ts`)

The previously-skipped "column alignment across rows" check is now implemented in a constrained form:

- **Only fires when two adjacent rows have the same visual count (N ≥ 2).** Rows with different counts (5 cards over 3 details) can't share a column grid, so they're ignored.
- **Only fires when the rows are vertically adjacent** (row B's top is within `gap + 10` of row A's bottom). Non-adjacent rows can legitimately use different grids.
- **Tolerance: 1px.** Rounding noise from equal-split math doesn't trigger false positives.
- **Severity: warning, not error.** `report.ok` stays `true`; the drift shows up in `report.issues` as `COLUMN_MISALIGN`.

Output example:
```
! [COLUMN_MISALIGN] Column 1: Row1 Col1 (x=15, w=414) does not align with
  Row2 Col1 (x=18, w=411) in the next row. Same-count adjacent rows should
  share a column grid.
```

New negative test case `BAD 8 — Column drift between same-count rows` added to `scripts/test-wireframe-validator.js`. The test harness gained a `runExpectCode()` variant because warnings don't flip `report.ok` — the new case asserts the `COLUMN_MISALIGN` code is present instead of asserting FAIL. Suite result: **13/13** (was 12/12).

All 5 canonical layouts (A–E) and all 7 pre-existing negative cases still pass unchanged.

## New and expanded skill docs

### New: `skills/calculations.md`

Previously there was no skill file for visual calculations — the code lived parked in `src/tools/calculations.ts` and the only reference was a bug entry in `tests.md`. An agent asked to add a "running total" or "rank" column had no pointer telling it (a) the MCP doesn't support visual calculations and (b) the right answer is usually a model-level measure anyway.

The new file covers:
- **What visual calculations are** and the `NativeVisualCalculation` PBIR format (so a contributor coming back to this can see the shape without reading the parked source)
- **Why they're parked** — writing the JSON loads cleanly but doesn't render; requires PBI Desktop internal state that file manipulation can't reproduce
- **Where DAX actually belongs** — the sibling `powerbi-modeling-mcp` (Microsoft's official MCP, ships its own skill docs) is the authority for anything model-level
- **External DAX references** — daxpatterns.com, dax.guide, Microsoft DAX reference
- **What to do if a user asks for a visual calculation** — 4-step fallback path ending in "refer to PBI Desktop manual entry"

### New: `skills/themes-per-visual.md`

`skills/themes.md` covered the five theme tools (`set_report_theme`, `get_report_theme`, `diff_report_theme`, etc.) but had one paragraph on the `visualStyles` block — the part of theme JSON an agent actually needs when the user says "all tables should use 9pt Segoe UI with alternating rows".

The new file is a ~250-line per-visual-type reference built from Microsoft's public theme schema. It covers:
- **Mental model** — two-tier structure (report-wide defaults + per-visual overrides), wildcard keys, arrays-not-objects rule
- **Property shapes** — why `"color": "#3B82F6"` silently fails and `"color": { "solid": { "color": "#3B82F6" } }` works
- **Category catalog by visual family** — universal (`title`, `background`, `border`, ...), charts (`categoryAxis`, `valueAxis`, `legend`, `labels`, `dataPoint`, ...), combo charts (`secondaryValueAxis`, `lineLabels`), tables/matrix (`grid`, `columnHeaders`, `values`, `total`, `subTotals`), cards (per-type because `card` / `cardVisual` / `kpi` use different category names), slicers, maps
- **Four worked examples** — universal title font, table formatting, column chart rule (with the combo-cascade gotcha), KPI with green trend
- **Workflow** — start from existing theme → build `visualStyles` → apply with `set_report_theme` → preview with `diff_report_theme` → verify in PBI Desktop
- **Gotchas** — array wrapping, color wrapping, case-sensitivity, silent unknown-property drops, classic vs new card name divergence, combo chart `secondaryValueAxis`, themes vs inline format precedence

Content is sourced from Microsoft's published theme schema — no proprietary material. Links out to `learn.microsoft.com/power-bi/create-reports/desktop-report-themes` for the canonical spec.

### Expanded: `skills/svg-visuals.md`

New "Where the DAX Actually Lives" section at the end. Makes the architectural boundary explicit: this MCP binds measures into visuals but doesn't author model-level DAX. The primary path for SVG measures is `powerbi-modeling-mcp → measure_operations(Create)` with `dataType: "String"`, `dataCategory: "ImageUrl"`. The fallback is `manage_extension_measures(add)` when the measure must be report-scoped. Ends with canonical DAX references (daxpatterns.com, dax.guide, Microsoft) so an agent writing SVG construction by hand knows where to look things up.

### Expanded: `skills/report-design.md` mental-model prelude

New 5-step "Mental Model — Before You Place A Single Visual" section at the top of the file, above the existing Layout Standards table. Five steps:

1. **Name the page's one job** — every page answers one question; two-question pages become two pages
2. **Pick the headline number** — one most-important metric, largest/leftmost/top-most card; everything else is context
3. **Context before detail** — headline → trend row → breakdown → detail table, expressed as the 3-30-300 rule
4. **Delete rather than add** — default action on an unclear visual is remove; if the user complains, you've learned it matters
5. **Pre-attentive attributes are a budget** — position, size, color, motion — one of each per page; if everything is red nothing is urgent

Frames the existing rules (5-7 colors, WCAG contrast, KPI card pattern, slicer placement) as consequences of the mental model rather than arbitrary constraints. Cites Few / Nussbaumer Knaflic / Tufte / IBCS as the canonical references — these are the common core of every dataviz book, not proprietary material.

## What Changes for Users

- **Agents** asked to delete/format/rebind more than 5 visuals in one call will now receive a safety-gate error on the first attempt and must explicitly set `confirmBulk: true` to proceed. Legitimate large operations still work; accidental page-wipes are caught.
- **Wireframe validation** now surfaces a warning when two adjacent rows of the same visual count drift horizontally. The warning is informational — `report.ok` stays `true` — but it points at the specific column and both offending visuals, making the drift easy to fix.
- **Skill coverage for DAX questions** is now explicit about the boundary between report MCP (binds measures) and modeling MCP (authors measures). Agents asked to write a running-total measure will be pointed at the right server instead of silently trying to write a visual calculation.
- **Theme requests** targeting a specific visual type (e.g. "all tables look like X") now have a dedicated reference file with the property shapes and category catalog, instead of the agent having to guess from the one-paragraph `visualStyles` mention in `skills/themes.md`.
- **Report design** now has a mental-model section that an agent loads before placing visuals. The 5 steps are short enough to hold in working memory but substantive enough to prevent "place 14 cards because the user asked for a dashboard" failures.
