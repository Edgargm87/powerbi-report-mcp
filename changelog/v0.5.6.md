# v0.5.6 — Skills Plumbing & Coverage Audit

**Date:** 2026-04-15

The first half of a two-step parity push. v0.5.6 wires the `skills/` directory into the runtime as a single source of truth and adds an automated audit that surfaces every tool with no skill coverage. v0.5.7 will use the audit to rewrite the 8 stale skills against the current tool surface.

## The Problem

Two parallel knowledge stores had drifted apart:

- **Tools** (~55) — TypeScript registrations under `src/tools/*.ts`, the executable surface
- **Skills** (10 markdown files under `skills/`) — prose that teaches LLM agents how to wield those tools

The skills directory was orphaned: nothing in `src/` ever read it. The `guide` tool's only two topics (`svg-visuals`, `report-design`) lived as ~250 lines of embedded TypeScript string constants in `src/tools/guide.ts`. Editing them required a TypeScript rebuild. Eight of the ten skills hadn't been touched since v0.4.x and described tools that no longer existed in their current form.

This release fixes the **plumbing** half of that gap. v0.5.7 will fix the **content** half.

## `guide` Tool Now Reads `skills/*.md` at Runtime

`src/tools/guide.ts` shrank from 337 lines to ~95. The two embedded constants moved to:

- `skills/svg-visuals.md` (was `SVG_VISUALS_GUIDE` const)
- `skills/report-design.md` (was `REPORT_DESIGN_GUIDE` const)

The tool now resolves `<projectRoot>/skills/` via `path.join(__dirname, "..", "..")` (same pattern `model-usage.ts` already uses for its `.usage/` cache) and:

- **Discovers topics dynamically** from the directory listing — adding a new `skills/*.md` file is enough, no code change required
- **Reads content live on every call** so editing a skill takes effect without restart
- **Strips the leading `<!-- doc-version -->` HTML comment** from output
- **Rejects path traversal** — topic key must match `/^[a-zA-Z0-9_-]+$/`
- **Supports a `list` topic** to query the current topic set: `guide({ topic: "list" })`

The tool description is built once at registration time from the directory snapshot. Editing skills mid-session updates content immediately but the description list updates on next server start — a fair trade for not re-globbing on every call.

## New: `scripts/audit-skill-coverage.js`

A standalone Node script that cross-references every registered MCP tool against every skill file and prints what's missing.

```bash
node scripts/audit-skill-coverage.js          # human-readable table
node scripts/audit-skill-coverage.js --json   # machine-readable
node scripts/audit-skill-coverage.js --strict # exit 1 if any gap (CI gate)
```

**Detection rules:**

- **Tool name** — regex-scans `src/tools/*.ts`, `src/model-usage.ts`, and `src/index.ts` for `server.tool("name", …)` and `_tool("name", …)` registrations
- **Parked tools excluded** — checks each file's exported `register*` function and skips it if `index.ts` doesn't actually invoke that function (filters out `calculations.ts`'s 3 parked tools)
- **Skill mention** — requires a backtick-wrapped occurrence of the tool name in any `skills/*.md` (e.g. `` `add_visual` ``, `` `add_visual(` ``). The backtick requirement keeps casual prose mentions ("the visual list") from masking real coverage gaps

**Baseline at v0.5.6:**

```
Tools registered:   55
With skill mention: 38
Missing coverage:   17
Coverage:           69.1%
```

## What Changes for Users

Nothing breaks. The `guide` tool still answers `guide("svg-visuals")` and `guide("report-design")` exactly as before. Two new affordances:

- `guide("list")` returns the current topic list (handy when adding new skills)
- Editing `skills/*.md` no longer requires a `npm run build`

## What's Next (v0.5.7)

Use the audit baseline to rewrite the 8 stale skills against the current tool surface. Each skill rewrite ends with `node scripts/audit-skill-coverage.js --strict` reporting fewer gaps than the previous commit. Target: 0 missing tools (100% coverage). Then add a CONTRIBUTING.md rule plus pre-commit/CI hook running `--strict`, so any new tool that ships without a matching skill mention fails the build.
