# v0.5.5 — Slicer Multi-Select & `reload_report` Default-Loaded

**Date:** 2026-04-15

Two small but high-impact fixes that came out of testing the `slicer test` and `chart test` reports.

## Slicer Multi-Select — Set & Detect

`add_visual` and `get_visual` now understand the `objects.selection` branch of slicer visuals, which is where Power BI Desktop persists the single-select / multi-select toggle.

### New `multiSelect` parameter on `add_visual`

```js
add_visual({
  visualType: "slicer",
  slicerMode: "Dropdown",
  multiSelect: true,                    // ← new
  bindings: [{ bucket: "Values", fields: [{ field: "financials[Country]", type: "column" }] }],
  ...
})
```

- `multiSelect: true`  → writes `objects.selection[0].properties.singleSelect = false` (checkbox / multi)
- `multiSelect: false` → writes `singleSelect = true` (radio / single)
- Omitted → keeps the prior PBI-default behaviour (Dropdown gets `strictSingleSelect:true`, Basic gets nothing → multi by PBI default)

Applies to both `slicer` and `listSlicer` visual types. `textSlicer` and `advancedSlicerVisual` do not have a single/multi toggle.

### `get_visual` slim mode now surfaces the state

```js
get_visual({ pageId, visualId })
// → {
//     ...,
//     slicerMode: "Dropdown",     // (slicer type only)
//     multiSelect: true            // (any slicer type)
//   }
```

Detection rules:

| Source | Result |
|---|---|
| `objects.selection[0].properties.singleSelect.expr.Literal.Value === "false"` | `multiSelect: true` |
| `... === "true"` | `multiSelect: false` |
| Property absent + `slicer` mode = `Dropdown` | `multiSelect: false` (PBI default) |
| Property absent + `slicer` mode = `Basic` | `multiSelect: true` (PBI default) |
| Property absent + `listSlicer` | `multiSelect: true` (PBI default) |

`slicerMode` is also extracted from `objects.data[0].properties.mode.expr.Literal.Value` (defaulting to `"Dropdown"` for the standard `slicer` type when absent).

### Why this matters

Before v0.5.5 the only way to toggle multi-select was a hand-rolled `format_visual` call passing the `selection` category with raw boolean literals — the agent had to know the PBIR object name and that its property was the inverted-meaning `singleSelect`. Detection was impossible from `get_visual` slim mode (the property was hidden in the full JSON). Both gaps are closed.

## `reload_report` Default-Loaded

`reload_report` (closes and reopens the `.pbip` in Power BI Desktop) is now in the `DEFAULT_TOOLS` set instead of the on-demand pool.

**Why:** the on-demand `load_tools` mechanism activates a tool server-side, but most LLM harnesses snapshot the MCP tool catalog at session start. A lazy-loaded `reload_report` could be activated successfully but never invoked because the harness's tool index didn't see it. Defaulting it sidesteps the trap entirely — and `reload_report` is something agents need on essentially every modeling-driven session, so the token cost is well spent.

`DEFAULT_TOOLS` count: 11 → 12.
