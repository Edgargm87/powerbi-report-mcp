# v0.5.4 — Calculation Groups Tab & Light Mode

**Date:** 2026-04-14

## Calculation Groups Tab

The usage dashboard now parses and displays calculation groups from both TMDL and BIM models.

- New **Calc Groups** tab between Functions and Pages, with a badge showing the count
- Each group renders as a collapsible page-card showing:
  - Group name and description
  - **Items** count and **Precedence** value
  - Per-item cards with ordinal, name, `///` TMDL description, and the full DAX expression
  - Format string expressions shown as inline `fmt:` badges when present
- TMDL parser handles multi-line calc item expressions (depth-3 continuation lines)
- The implicit `Name` column of a calc group table is correctly excluded from the Columns tab

## Light Mode

The dashboard now supports a light theme alongside the default dark theme.

- **Toggle button** (`☾` / `☀`) in the header, next to Refresh
- **Persistence** via `localStorage['usage-theme']` — survives reloads and auto-refresh cycles
- **No FOUC** — an inline `<script>` in `<head>` sets `data-theme` before the body paints
- Theme is driven by CSS custom properties in `:root` and `[data-theme="light"]`:
  - Backgrounds (`--bg`, `--surface`, `--surface-alt`, `--surface-center`)
  - Borders (`--border`, `--border-soft`, `--border-row`)
  - Text (`--text`, `--text-body`, `--text-muted`, `--text-dim`, `--text-faint`, `--text-fainter`)
  - Chip palettes (`--chip-dep-*`, `--chip-used-*`)
  - Syntax highlighting (`--code-name`, `--code-type`, `--code-punct`) for UDF parameter signatures
- All inline JS styles that hardcoded dark-specific colors were moved to classes or switched to `var(--…)` so both themes render correctly

## KPI Tooltips

Hover any summary tile to see a custom tooltip explaining what it means:

- **Direct** — "Fields bound to at least one visual…"
- **Indirect** — "Not on any visual, but referenced by direct measures via DAX…"
- **Unused** — "Not referenced anywhere in the report — safe to remove"
- Tooltips explain the `NM · NC` shorthand (measures vs columns)
- First and last tiles align their tooltips to the grid edge to avoid viewport clipping
- Uses `.has-tip[data-tooltip]` with `::before` / `::after` pseudo-elements — no JS tooltip library

## Copy-DAX Button

Every DAX code block (`.lineage-dax`) gets a small `⎘` button in the top-right, visible on hover.

- Used on measure lineage expressions, UDF function bodies, and calc group item expressions
- Click copies the raw DAX to the clipboard — no selection needed
- Confirms with a green `✓` for 1.5s, then resets
- Graceful fallback: tries `navigator.clipboard.writeText` first, falls back to a hidden-textarea + `execCommand('copy')` for older browsers
- `addCopyButtons()` is called on init and after every `openLineage()` call; idempotent via a `data-copy-wired` marker so re-renders are safe

## Hidden-Page Indicator

Pages marked `HiddenInViewMode` in their `page.json` (tooltip pages, drillthrough targets, nav-suppressed pages) now show a purple **HIDDEN** badge next to the page name in the Pages tab.

- `scanReportBindings` collects hidden page display names in a dedicated `hiddenPages: string[]` list
- `FullData` exposes `hiddenPages` to the client-side JS as `DATA.hiddenPages`
- `renderPages()` builds a `Set` once and injects the badge in each page-header when the page name matches
- Badge styled via `.hidden-badge` with its own light-mode override (darker purple on a tinted background)
- Hover tooltip (`title` attribute) explains what `HiddenInViewMode` means
- The MCP tool JSON response (both `slim` and `full`) now includes `hidden: boolean` on each page entry plus a top-level `hiddenPages` array, so agents can reason about tooltip/drillthrough pages without re-parsing the report

## Wireframe Validator

New pure-function module `src/wireframe-validator.ts` that statically verifies any list of visual specs against the canvas layout rules in `docs/wireframes.md`.

Checks:

| Code                  | Catches |
|-----------------------|---------|
| `OUT_OF_BOUNDS`       | Negative x/y, right edge > 1280, bottom edge > 720 |
| `LEFT_MARGIN`         | Non-banner visual at x < 20 |
| `RIGHT_MARGIN`        | Non-banner right edge > 1260 (e.g. 245x5 card rounding) |
| `BOTTOM_MARGIN`       | Non-banner bottom edge > 714 (6px bottom breathing room) |
| `WRONG_GAP_H` / `_V`  | Horizontal / vertical gap != 5px between neighbours |
| `OVERLAP`             | Any two visuals with intersecting bounding boxes |
| `SILENT_DEFAULT`      | Non-banner visual at (0,0) — the classic "forgot x/y" bug |
| `BANNER_POSITION`     | Banner height != 52px |
| `NEGATIVE_DIMENSION`  | width or height <= 0 |

Row and column groups are auto-detected by geometric overlap — the caller doesn't have to label the intended layout. Returns a structured `WireframeReport` with stats (coverage %, rightmost edge, bottom edge, error/warning counts) and a `formatReport()` pretty-printer.

### 6px Bottom Margin

`CANVAS.marginBottom` is now **6px** (was 0), so non-banner visuals must land at `y + height ≤ 714`. Purpose: keep the last row from sitting flush against the bottom of the canvas. This adds a new `BOTTOM_MARGIN` error code and shrinks the heights in all five validated layouts accordingly:

| Layout | Before | After | Bottom |
|--------|--------|-------|--------|
| A — Dashboard | details h=278 | h=277 | 714 |
| B — Analysis  | table   h=233 | h=227 | 714 |
| C — KPI Summary | chart h=413 | h=407 | 714 |
| D — Sidebar Nav | nav h=663, detail h=278 | nav h=657, detail h=277 | 714 |
| E — 3×3 Tiles | tiles h=217 (rows 57/279/501) | h=215 (rows 57/277/497) | 712 |

Wire E lands at 712 rather than 714 because 3 uniform rows can't divide 657 usable px exactly; the extra 2px end up as additional bottom breathing room.

## Skill Doc Rewrite (skills/wireframes.md v2.0)

Before v2.0 the skill doc used **10px margins and 10px gaps**, with a fake "header bar" at `x:10 y:10 w:1260 h:40` instead of a proper banner. Running the validator against its 15 layouts produced **14-22 errors each** — every layout taught the LLM the wrong math.

- Rewrote the skill doc from scratch with the canonical `docs/wireframes.md` constants (20px margins, 5px gaps, banner at `x:0 y:0 w:1280 h:52`)
- Replaced the 15 broken layouts with 5 validated layouts:
  - **A** Dashboard (5 KPIs + 2 charts + 3 details) — 11 visuals, 93.8% coverage
  - **B** Analysis (3 slicers + 2/3+1/3 chart/KPI split + full-width table) — 10 visuals, 94.2%
  - **C** KPI Summary (6 cards + wide chart) — 8 visuals, 94.8%
  - **D** Sidebar Nav (160px rail + 4 KPIs + 2 charts + table) — 9 visuals, 94%
  - **E** 3x3 Tile Grid (9 uniform tiles) — 10 visuals, 94.1%
- Added a "Common Mistakes" table mapping each misalignment cause to the validator error code it produces, and a section showing how to call the validator from Node or TypeScript.
- Added explicit warnings against relying on the `x=0, y=0` default in the `add_visual` Zod schema.

## Test Runner

New `scripts/test-wireframe-validator.js` runs the validator against:
- The 5 canonical wireframe layouts A–E (positive cases)
- 7 deliberately-broken layouts demonstrating each misalignment root cause (rounding overflow, 10px gap, missing margin, overlapping visuals, silent (0,0) default, bottom-edge overflow past 720, bottom-margin violation 718 > 714)

Result: **12/12 tests match expectation**, all canonical layouts pass with 0 errors / 0 warnings.

Also added `scripts/test-skill-layouts.js` as a historical diagnostic that replays the pre-v2.0 skill layouts against the validator — useful if the skill doc ever drifts again.

## Shape Text in `objects.text`

`src/helpers/createVisual.ts` previously emitted shape labels into the `general.paragraphs` branch (textbox schema). PBIR accepted the JSON, but Power BI Desktop silently ignored the text — shapes rendered blank.

- Rewritten to emit the two-entry `visual.objects.text` array that Power BI Desktop actually reads: entry `[0]` is the `show: true` toggle, entry `[1]` carries the real properties with `selector: { id: "default" }`
- New `createVisual` options: `textVAlign`, `textPadding`, `textItalic`, `textUnderline`, `textFont`, alongside the existing `textContent`, `textColor`, `textAlign`, `textBold`, `textSize`
- Default font color is emitted as `ThemeDataColor { ColorId: 1 }` so labels auto-invert when the theme changes (matches Power BI Desktop's default)
- New `POWER_BI_FONT_STACKS` map with 21 friendly font names (Segoe UI variants, DIN, Arial, Calibri, Georgia, Times New Roman, Consolas, …) that expand to the full PBIR CSS font stack with correct DAX quote escaping. Unknown values are passed through verbatim as escape-hatch stacks.
- DAX literal suffixes are now correct: `L` for int64 margins, `D` for double font size, raw `true`/`false` for booleans
- Single quotes inside label text are auto-doubled (`Jonny's` → `Jonny''s`)

## Wireframe Test Builder

New `scripts/build-wireframe-test.py` (re)builds all five wire pages (A–E) in `pbi report/wireframe test.Report` from a single Python source, with shape text labels inside each shape describing the visual type the space represents.

- Idempotent: scans `pages/` for pages whose displayName starts with `Wire ` and reuses those IDs, so pages.json never churns
- Purges each page's `visuals/` dir before rewriting so stale placements from prior runs don't linger (tolerates OneDrive directory locks by continuing past `WinError 5`)
- Emits the full PBIR shape schema directly: shape/rotation/fill/outline/text objects, with the validated 6px-bottom-margin geometry
- Banner uses Segoe UI Bold 14pt white on navy; tile labels use Segoe UI Semibold 11pt bold — all via the `font_family_literal()` helper that mirrors the map in `createVisual.ts`
