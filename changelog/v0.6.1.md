# v0.6.1 — Binding Validation

**Date:** 2026-04-15

v0.6.1 closes the single biggest silent-failure hole in the bind path: when an agent binds `Sales[FooBar]` and `FooBar` doesn't exist in the semantic model, the visual used to load into PBI Desktop and render nothing — the mistake wasn't noticed until someone opened the report. `model_usage` already had the data to catch this; we just weren't running the check on bind. Now we are.

All three bind paths (`add_visual`, `update_visual_bindings`, `bulk_bind`) run every `FieldSpecInput` through a validator before any write. Unknown fields fail the call with nearest-match suggestions instead of producing a broken visual.

## Behaviour changes

### Binding validation on `add_visual` / `update_visual_bindings` / `bulk_bind`

Every `bindings` array is checked against the semantic model's field inventory (same source as `model_usage`) before the write. The validator reads the sibling `.SemanticModel` folder, builds a per-table inventory of columns + measures, merges in extension measures from `reportExtensions.json`, exposes calc-group items as measures, and matches each field spec against it.

**Three modes**, controlled per-call by `strictBindings` or globally by `MCP_BINDING_VALIDATION`:

| Mode | Per-call | Env var | On unknown field |
|------|----------|---------|------------------|
| **strict** (default) | `strictBindings: true` | `MCP_BINDING_VALIDATION=strict` | Call errors out, nothing written |
| **warn** | `strictBindings: false` | `MCP_BINDING_VALIDATION=warn` | Call proceeds, errors attached as `bindingWarnings` |
| **off** | — | `MCP_BINDING_VALIDATION=off` | Validation skipped entirely |

Precedence: per-call param beats env var beats default. Default is strict.

**Silent degradation when the model is unreadable.** If the sibling `.SemanticModel` folder can't be located (live-connect, missing folder, parse error), validation is skipped silently and the call proceeds. The validator must never block a legitimate workflow just because the model isn't there.

**Error reasons** — every validation error carries a stable `reason` code:
- `table_not_found` — entity name doesn't match any table
- `column_not_found` / `measure_not_found` — field not in the target table for its kind
- `type_mismatch_column_is_measure` — spec says `type: "column"` but it's actually a measure
- `type_mismatch_measure_is_column` — spec says `type: "measure"` but it's a column (use `type: "aggregation"`)
- `parse_error` — field shorthand doesn't match `Table[Column]` pattern

**"Did you mean" suggestions.** Each error carries up to 3 nearest-match suggestions, already formatted as `Table[Field]`. Matching is case-insensitive Levenshtein with a length-bounded cap — so a typo like `Sales[Qantity]` suggests `Sales[Quantity]`, and `sales[orderdate]` suggests `Sales[OrderDate]`. Unrelated names return zero suggestions (the goal is typo rescue, not arbitrary-name guessing).

**Example strict-mode error response:**
```json
{
  "success": false,
  "error": "Binding validation failed (2 issues):\n  • Sales[FooBar] (column): column not found in table 'Sales'. Did you mean: Sales[Quantity]?\n  • Slaes[Total Sales] (measure): table 'Slaes' not found in model. Did you mean: Sales[Total Sales]?\nValidation runs against the sibling .SemanticModel folder + report extension measures. To bypass for a single call set strictBindings: false, or set MCP_BINDING_VALIDATION=off globally.",
  "bindingErrors": [ /* structured per-error objects with reason, label, suggestions */ ],
  "mode": "strict"
}
```

**Case-sensitivity.** Field names are matched exactly — `sales[quantity]` is treated as `table_not_found` because PBI Desktop is case-sensitive here too. The casing mistake still surfaces the correctly-cased name as the top suggestion.

### New model-usage export: `getModelFieldInventory(reportPath)`

The validator reads its field inventory through a new exported helper in `src/model-usage.ts`. Shape:

```typescript
export interface ModelFieldInventory {
  tables: Map<string, { columns: Set<string>; measures: Set<string> }>;
  tableNames: string[];
  extensionMeasures: Map<string, Set<string>>;
  builtAt: number;
}

export function getModelFieldInventory(reportPath: string): ModelFieldInventory | null;
```

Returns `null` when the model can't be located or parsed (silent degrade). Cached in-process; cache is cleared by the existing `invalidateCache()` entry point whenever the report is written, so the inventory stays fresh across bind calls.

## New helper module: `src/helpers/bindingValidation.ts`

~300 lines, pure. Exports:

- `ValidationMode = "strict" | "warn" | "off"`
- `BindingValidationError` — structured error with `reason`, `label`, `entity`, `property`, `kind`, `suggestions`, optional `rawMessage`
- `ValidationOutcome` — `{ proceed, errors, message, mode }` returned by the one-call helper
- `resolveValidationMode(strictBindings)` — applies per-call > env > default precedence
- `getInventoryForProject(project, mode)` — returns `null` for `off` or when inventory is unavailable
- `validateFieldSpec(spec, inventory)` — single field validation
- `validateBindings(bindings, inventory)` — batch validation
- `formatBindingErrors(errors)` — human-readable output with "did you mean" suggestions
- `runBindingValidation(project, bindings, strictBindings)` — one-call wrapper used by every tool handler

The three tool handlers don't call `validateFieldSpec` directly — they call `runBindingValidation`, which owns the mode resolution, inventory lookup, and short-circuit logic in one place. One choke point for every bind path.

## New test script + npm gate

`scripts/test-binding-validator.js` runs 25 assertions against an in-memory `ModelFieldInventory` — no PBI Desktop required, no sibling `.SemanticModel` folder required. Coverage:

- **6 positive cases** — column / measure / aggregation shorthand, entity+property split, extension measure, measure-as-aggregation
- **7 negative cases** — every `reason` code exercised end-to-end
- **3 suggestion quality checks** — typo rescue (`Qantity` → `Quantity`), measure-scope suggestions, unrelated-name zero-suggestion
- **3 batch checks** — multi-bucket error accumulation, null-inventory silent degrade, empty-bindings no-op
- **5 mode resolution checks** — default, env=off/warn, per-call overrides env

Wired into `npm run test:all` alongside the wireframe validator. New `test:binding` npm script runs it standalone.

## What Changes for Users

- **Agents** that typo a field name — `Sales[Qantity]` instead of `Sales[Quantity]`, `Slaes[Total Sales]` instead of `Sales[Total Sales]` — now get a structured error with the correct name suggested, instead of a silently-broken visual that only surfaces when a human opens PBI Desktop. First-call failure is cheap; discovery after the fact is not.
- **Live-connect reports** and reports without a sibling `.SemanticModel` folder behave exactly as before. Validation degrades silently when it has nothing to validate against.
- **Model/report out-of-sync cases** — e.g. the agent is about to add a measure via `powerbi-modeling-mcp` and needs to bind it in this MCP before the model write lands — can set `strictBindings: false` on the single call to let it through as a warning. Don't flip `MCP_BINDING_VALIDATION=off` globally; you'd lose the safety net everywhere.
- **Batch calls** (`add_visual` with `visuals: [...]`, `bulk_bind` with `updates: [...]`) validate the entire flattened binding set in one pass. A single unknown field in the batch fails the whole call in strict mode — atomic, no partial writes.
