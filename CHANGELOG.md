| [**0.9.3**](changelog/v0.9.3.md) | 2026-05-04 | **Bug fix: measure home-table auto-resolution.** `pbir_bulk_bind` / `pbir_add_visual` / `pbir_update_visual_bindings` used to silently produce broken visuals when the LLM specified a fact-table entity for a measure that lived elsewhere (e.g. `Sales[Total Revenue]` when the measure was authored on `_Measures`) — Power BI Desktop accepted the file, the visual showed the right field name, but no data rendered because `SourceRef.Entity` pointed at the wrong table. Two halves of the fix: (1) `validateFieldSpec` now walks every other table when a measure isn't found in the specified entity and surfaces home-table matches as top-priority suggestions (`(actual home table)` when single, `(candidate home table)` when ambiguous). Ambiguous cases never auto-correct — they surface as a normal `measure_not_found` error with both candidates listed. Columns and aggregations are never auto-corrected. Wired through every binding code path: `pbir_add_visual` (visuals.ts), `pbir_update_visual_bindings` (bindings.ts), `pbir_bulk_bind` (bulk.ts, per-entry inventory in `continueOnError` mode), and `pbir_layout_grid` (layoutGrid.ts). Even with `MCP_BINDING_VALIDATION=off`, the inventory still loads so the auto-resolver runs (silent-correct is the spirit of "off"). When the model can't be located (live-connect, missing sibling `.SemanticModel`), behaviour is unchanged. New `scripts/test-measure-home-table.js` (7 assertions) wired into `test:all` as `test:measure-home-table`. `skills/visuals.md` gains a "Measure home table — auto-resolution" subsection.
